# Flowchart

**Use when**: explaining a process, decision tree, system architecture, or workflow with conditional branches — the structure and sequence of steps is the message.  
**Avoid when**: the goal is to compare quantities or show trends; the process has more than ~15–20 nodes (use a simplified summary diagram instead).

## Key rules

- Use standard shapes consistently: rectangles for process steps, diamonds for decisions, ovals/rounded rects for start/end
- Flow direction: top-to-bottom or left-to-right — never both in the same diagram
- Label every decision branch (Yes/No, True/False, or the condition text)
- Minimize crossing arrows — rearrange nodes to reduce visual noise
- Keep labels short (verb + noun): "Validate input", not "The system performs validation of the input data"

## Anti-patterns

- Inconsistent shape usage — mixing rectangles and diamonds arbitrarily confuses readers
- Unlabeled decision branches — readers can't tell which path is which
- Overly detailed flowchart for a high-level audience — abstract to the level the audience needs
