# Issue Definition Process

Use this sequence to turn a request into an implementation-ready issue.

1. **Classify the request**
   - Surface: Frontend, Backend, or Fullstack.
   - Work type: Feature, Bugfix, Refactor, Improvement, or Research-backed change.

2. **Pick the issue shape**
   - Start with the required core sections from `assets/issue-template.md`.
   - Add optional sections only when they materially change implementation or validation.

3. **Frame the issue**
   - Write a precise title.
   - State the problem or request in observable terms.
   - Add the goal, and add why now only if it is not already obvious.

4. **Set boundaries**
   - List scope and non-goals.
   - Capture assumptions, dependencies, and constraints only where they matter.

5. **Write requirements and applicable fork details**
   - Functional requirements first.
   - Non-functional requirements only when material.
   - Add only the relevant surface details and work-type details.

6. **Define completion**
   - Add acceptance criteria.
   - Add readiness, done, and validation expectations when they help remove ambiguity or risk.

7. **Slice and expose gaps**
   - Break the work into small implementation steps.
   - Put unresolved material gaps under assumptions, open questions, or Definition of Ready.
